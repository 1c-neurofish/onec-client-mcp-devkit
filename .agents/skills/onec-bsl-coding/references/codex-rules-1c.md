# Codex Rules для разработки на 1С

Набор правил основан на принципах проектирования:

- SRP — Single Responsibility Principle  
- OCP — Open/Closed Principle  
- DRY — Don't Repeat Yourself  
- YAGNI — You Aren’t Gonna Need It  
- Separation of Concerns  
- High Cohesion / Low Coupling  
- Law of Demeter  
- Composition over Inheritance  

Правила предназначены для использования Codex при генерации и анализе кода 1С.

---

# SRP — Single Responsibility Principle

**Каждый модуль, функция и процедура должны иметь одну ответственность.**

### Rules

- Каждый метод выполняет **одну логическую операцию**.
- Метод не должен одновременно:
  - выполнять бизнес-логику
  - работать с базой данных
  - формировать UI сообщения.
- Если метод начинает делать более одной задачи — его необходимо **разделить**.
- Общие модули должны иметь **четко определенную ответственность**:
  - расчеты
  - работа с данными
  - интеграции
  - утилиты.
- Процедуры длиннее **40–60 строк** считаются кандидатами на декомпозицию.

---

# OCP — Open/Closed Principle

**Код должен быть открыт для расширения и закрыт для изменения.**

### Rules

- Новая функциональность должна добавляться **через расширение**, а не через изменение существующего кода.
- Избегать длинных конструкций:

```

Если Тип = ...
ИначеЕсли Тип = ...
ИначеЕсли Тип = ...

```

- Для вариативного поведения использовать:
  - стратегии
  - обработчики
  - таблицы соответствий
  - подписки на события.
- Любая логика, зависящая от типа документа/операции/режима, должна иметь **точку расширения**.

---

# DRY — Don't Repeat Yourself

**Одна бизнес-логика должна иметь одно место реализации.**

### Rules

- Повторяющиеся блоки кода должны выноситься в **общие функции**.
- Запрещено копировать одинаковые расчеты или проверки.
- Повторяющиеся тексты сообщений должны быть вынесены в **константы или функции**.
- Повторяющиеся запросы должны формироваться через **единый метод построения запроса**.
- Любое повторение логики более **двух раз** требует рефакторинга.

---

# YAGNI — You Aren’t Gonna Need It

**Не реализовывать функциональность заранее.**

### Rules

- Не добавлять параметры "на будущее".
- Не реализовывать универсальные механизмы без реального сценария использования.
- Не создавать абстракции без минимум **двух реальных вариантов использования**.
- Избегать преждевременного усложнения архитектуры.

---

# Separation of Concerns

**Разделение ответственности между слоями системы.**

### Rules

- UI не должен содержать:
  - бизнес-логику
  - SQL-запросы
  - транзакции
  - интеграции.
- Формы должны:
  - получать ввод пользователя
  - вызывать сервисные методы
  - отображать результат.
- Бизнес-логика должна располагаться в:
  - общих модулях
  - модулях объектов
  - сервисных модулях.
- Работа с БД должна быть изолирована в **репозиториях или сервисах доступа к данным**.

---

# High Cohesion / Low Coupling

**Высокая связность внутри модуля и низкая зависимость между модулями.**

### Rules

- Модуль должен содержать функции **одной предметной области**.
- Избегать модулей-свалок типа `ОбщийМодульУтилиты`.
- Модули должны взаимодействовать через **минимальный публичный API**.
- Избегать прямого доступа к внутренним структурам других модулей.

---

# Law of Demeter

**Метод должен знать только о своих непосредственных зависимостях.**

### Rules

- Избегать длинных цепочек вызовов:

```

Объект.Данные.Клиент.Адрес.Город

```

- Каждый объект должен предоставлять **собственные методы доступа**.
- Логика работы с внутренними структурами объекта должна находиться **внутри этого объекта**.

---

# Composition over Inheritance

**Предпочитать композицию наследованию.**

### Rules

- Поведение должно расширяться через **композицию объектов**, а не через сложные иерархии.
- Избегать глубокого наследования.
- Поведение должно передаваться через **внедрение зависимостей** (dependency injection).
- Использовать сервисные объекты вместо наследования логики.

---

# Общие рекомендации

- Код должен быть **простым и читаемым**.
- Предпочитать **маленькие функции**.
- Минимизировать побочные эффекты.
- Каждая функция должна иметь **понятное имя, отражающее действие**.
