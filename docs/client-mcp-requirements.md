# Требования к расширению client-mcp

Документ фиксирует целевые требования к расширению `exts/client-mcp`. Это требования к ядру MCP внутри клиента 1С, а не к конкретным прикладным MCP-провайдерам вроде `test_client`.

Связанный ADR: [ADR-0001](decisions/0001-client-mcp-provider-architecture.md)

## Назначение

- `client-mcp` реализует инфраструктуру и обвязку для создания MCP-сервера внутри клиента 1С.
- Ядро владеет транспортом, жизненным циклом MCP-сервера, реестрами, маршрутизацией протокольных вызовов, состоянием, ошибками и API завершения долгих операций.
- Прикладные MCP-сущности создаются отдельно от ядра: через внешние обработки, общие модули, отдельные расширения или другие подключаемые контуры 1С.
- Новые прикладные возможности не должны требовать правок транспорта или протокольного слоя `client-mcp`, если не меняется общий контракт ядра.

## Модель расширения

- Путь создания нового MCP должен быть максимально коротким.
- Целевой минимальный сценарий для прикладного разработчика - экспортная функция или процедура регистрации, которую runtime может обнаружить и вызвать.
- Регистрация должна поддерживать как autodiscovery-провайдеры, так и явную регистрацию из внешней обработки.
- Публичный API `client-mcp` должен скрывать JSON-RPC envelope, transport events и внутренние реестры от прикладного кода.
- Для входных данных публичного API не добавляется скрытая нормализация "на всякий случай"; если контракт неудобен или неверен, меняется контракт и его callers.

## Поддерживаемые MCP-сущности

`client-mcp` должен поддерживать регистрацию и выполнение:

- `tools`;
- `resources`;
- `prompts`.

Поддержка `tools`, `resources` и `prompts` уже является базовым контрактом ядра.

`tasks` и task-based lifecycle исключены из активной работы и считаются замороженным направлением. Наличие внутренних методов или черновиков для task не делает этот сценарий поддержанным публичным контрактом.

## Модель выполнения tool call

Tool call должен поддерживать два класса выполнения:

- синхронное выполнение: обработчик возвращает результат в рамках текущего вызова;
- асинхронное выполнение обычного `tools/call`: обработчик запускает долгую работу, возвращает отложенный ответ, а итоговый результат отправляется отдельно через API `client-mcp`.

Требования к долгим tool call:

- должны поддерживаться долгие операции, которые фактически выполняются дольше обычного клиентского timeout;
- синхронные долгие операции не должны приводить к молчаливому обрыву клиента без понятного статуса;
- отправка итогового ответа для асинхронного `tools/call` должна выполняться через публичный API `client-mcp`, а не прямой работой с транспортом;
- прогресс долгих операций должен рассматриваться как часть публичного контракта через MCP `notifications/progress`, если клиент передал `progressToken`.

## Требования к документации и проверке

- Для каждого поддержанного сценария расширения должен быть минимальный пример в документации.
- Для `tools`, `resources` и `prompts` должны быть отдельные приемочные сценарии или YAxUnit-тесты.
- Для долгих tool call должен быть проверен обычный `tools/call` с отложенной отправкой результата.
- Тесты должны проверять сам факт и формат `notifications/progress`, а не только наличие `progressToken` во входных данных.
- Нельзя считать поддержку progress завершенной без рабочего сценария, который проходит через реальный транспортный контур.
