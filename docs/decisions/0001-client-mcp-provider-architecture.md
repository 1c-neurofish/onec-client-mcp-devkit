# ADR-0001: Размещать MCP-сервер в клиентском расширении 1С и расширять его через провайдеры

- Статус: accepted
- Дата: 2026-04-22
- Связанные документы: [ARCHITECTURE.md](../../ARCHITECTURE.md), [Требования client-mcp](../client-mcp-requirements.md), [Контракты test client](../mcp-test-client/contracts.md), [Инварианты решения](../solution-invariants.md)

## Контекст

В репозитории уже реализован MCP-сервер, который запускается внутри клиентского контура 1С и работает через внешнюю компоненту `AddIn.WebTransport.mcp` из проекта [alkoleft/web-transport-addin](https://github.com/alkoleft/web-transport-addin).

Фактическая архитектура по коду выглядит так:

- Жизненный цикл приложения и точка входа находятся в `exts/client-mcp/src/Configuration/ManagedApplicationModule.bsl`: модуль приложения хранит только глобальную переменную состояния `Мсп_Состояние` и обработчики событий.
- Доступ к глобальному состоянию инкапсулирован в `exts/client-mcp/src/CommonModules/Мсп_СостояниеКлиент/Module.bsl`.
- Инициализация MCP-контура выполняется через `Мсп_Сервер.Инициализировать()`, который поднимает реестры и регистрирует провайдеров.
- Транспорт и разбор внешних событий отделены от протокольной логики: `Мсп_ТранспортСерверКлиент` управляет компонентой, `Мсп_ТранспортСобытияКлиент` принимает события `WebTransport`, `Мсп_ПротоколКлиент` формирует JSON-RPC/MCP-ответы и вызывает обработчики.
- Расширяемость построена на провайдерах: `Мсп_МетаданныеСервер` ищет общие модули в подсистеме `Мсп_Провайдеры` и подсистемах с суффиксом `_Мсп_Провайдеры`, а `Мсп_ПровайдерыКлиент` вызывает у найденных модулей экспортный метод `Зарегистрировать`.
- Возможности публикуются не напрямую из транспорта, а через общие реестры инструментов, ресурсов, шаблонов ресурсов и промптов в `Мсп_РеестрКлиент`.
- UI-автоматизация вынесена в отдельное расширение `exts/test_client`, которое публикует свои инструменты и ресурсы как провайдер, используя URI-контракт `window://`, `form://`, `control://`, `snapshot://`.

Отдельного каталога ADR в репозитории до этой задачи не было, поэтому это решение зафиксировано постфактум по текущему коду и тестам, а не по исторической переписке.

## Решение

Поддерживать и развивать текущую архитектурную схему:

1. MCP-сервер, транспорт, протокол, состояние и реестры остаются в клиентском расширении `exts/client-mcp`.
2. Доменные возможности публикуются через провайдеры, обнаруживаемые по метаданным, а не через прямое редактирование транспортного слоя.
3. Интеграция с тест-клиентом и UI-автоматизация остаются отдельным провайдерным контуром в `exts/test_client`.
4. Доступ к глобальному состоянию выполняется только через `Мсп_СостояниеКлиент`; модуль приложения не становится точкой для бизнес-логики.
5. Публичные контракты остаются строгими: режимы выполнения инструментов, форматы URI, структура ответов и коды ошибок не расширяются скрытой нормализацией "на всякий случай".
6. Task-based lifecycle исключен из активной архитектуры и считается замороженным направлением; асинхронный tool call трактуется как обычный `tools/call` с отложенным ответом.

## Рассмотренные альтернативы

### 1. Регистрировать все инструменты и ресурсы напрямую в ядре MCP

Такой подход упростил бы трассировку для небольшого MVP, но сломал бы текущую модульность. Любое расширение возможностей вынуждало бы править транспорт, протокол или общий модуль сервера вместо локального провайдера.

### 2. Вынести MCP-сервер из клиентского контура 1С

Это уменьшило бы зависимость от клиентского расширения и внешней компоненты, но разрушило бы текущий контракт взаимодействия с объектами UI и тест-клиентом, которые сейчас доступны внутри клиентского процесса и уже покрыты тестами.

### 3. Текущая схема: ядро в `client-mcp`, функциональность в провайдерах

Этот вариант уже реализован и подтвержден кодом и тестами. Он изолирует транспорт и протокол от доменных возможностей, позволяет отдельно развивать `test_client` и удерживает границы изменений узкими.

## Последствия

- Новые инструменты, ресурсы и промпты должны появляться через новые или существующие провайдеры, а не через разрастание `Мсп_Транспорт*` и `Мсп_ПротоколКлиент`.
- Архитектурные изменения в публичных контрактах теперь должны сопровождаться обновлением ADR и инвариантов, а не только изменениями в коде.
- Строгая валидация аргументов и URI сохраняет явность контракта, но означает, что при расхождении ожиданий callers будут получать `INVALID_PARAMS`, `NOT_FOUND`, `NOT_READY` или `ACTION_DENIED`, а не неявную адаптацию данных.
- Провайдеры изолируют точки расширения, но требуют дисциплины: сигнатура `Зарегистрировать` и поля дескрипторов должны соблюдаться точно.

## Не цели

- Этот ADR не фиксирует окончательную эволюцию MCP API или полное соответствие всем будущим версиям спецификации.
- Этот ADR не заменяет подробные контрактные документы для `test_client`.
- Этот ADR не утверждает, что все будущие провайдеры обязаны жить в `exts/test_client`; он фиксирует только текущий способ их подключения и границы ядра.

## План реализации для дальнейших изменений

- Затрагиваемые области:
  - `exts/client-mcp/src/Configuration/ManagedApplicationModule.bsl`
  - `exts/client-mcp/src/CommonModules/Мсп_СостояниеКлиент/Module.bsl`
  - `exts/client-mcp/src/CommonModules/Мсп_МетаданныеСервер/Module.bsl`
  - `exts/client-mcp/src/CommonModules/Мсп_ПровайдерыКлиент/Module.bsl`
  - `exts/client-mcp/src/CommonModules/Мсп_РеестрКлиент/Module.bsl`
  - `exts/client-mcp/src/CommonModules/Мсп_ПротоколКлиент/Module.bsl`
  - `exts/client-mcp/src/CommonModules/Мсп_ТранспортСерверКлиент/Module.bsl`
  - `exts/client-mcp/src/CommonModules/Мсп_ТранспортСобытияКлиент/Module.bsl`
  - `exts/test_client/src/CommonModules/*`
  - `tests/src/CommonModules/ОМ_*`

- Правила внесения изменений:
  - Новую функциональность публиковать через экспортный `Зарегистрировать(ИсточникРегистрации)` в провайдерном модуле, доступном через подсистему `Мсп_Провайдеры`.
  - Для регистрации использовать `Мсп_Сервер.НовыйИнструмент`, `Мсп_Сервер.НовыйРесурс`, `Мсп_Сервер.НовыйШаблонРесурса`, `Мсп_Сервер.НовыйПромпт` и не обходить проверки `Мсп_РеестрКлиент`.
  - Глобальное состояние читать и изменять только через `Мсп_СостояниеКлиент`.
  - Новые публичные URI, новые режимы выполнения инструментов, разморозку task-based lifecycle или изменение envelope ответов считать архитектурным изменением и оформлять отдельным ADR.
  - При изменении контрактов провайдера или протокола одновременно обновлять тесты `ОМ_Провайдеры`, `ОМ_РеестрКлиент`, `ОМ_ПротоколКлиент`, `ОМ_URIКлиент`, `ОМ_ФормыКлиент`, `ОМ_ОкнаКлиент`, `ОМ_УправлениеТестКлиентом`.

## Проверка

- [ ] Инициализация по-прежнему проходит через `Мсп_Сервер.Инициализировать()` и не дублируется в модуле приложения.
- [ ] Новые возможности публикуются через провайдерный модуль и попадают в `Мсп_РеестрКлиент`, а не регистрируются в транспорте вручную.
- [ ] Обработчики инструментов используют обычный `tools/call`; асинхронность реализуется только через отложенный ответ для `sync`, без активного task-based lifecycle.
- [ ] Контракты `window://`, `form://`, `control://`, `snapshot://` сохраняют совместимость либо оформлены отдельным ADR.
- [ ] Ошибки публичного API остаются явными и проверяются тестами, без скрытой автонормализации входа.
- [ ] Тесты `ОМ_Провайдеры`, `ОМ_ПротоколКлиент`, `ОМ_URIКлиент` и связанные с UI сценарии остаются согласованными с кодом.
