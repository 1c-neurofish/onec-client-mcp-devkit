# Инварианты решения

Документ фиксирует инварианты, извлеченные из текущего кода и тестов. Если какой-либо инвариант перестает быть верным, это должно сопровождаться осознанным изменением кода, тестов и, при необходимости, ADR-0001.

Связанный ADR: [ADR-0001](decisions/0001-client-mcp-provider-architecture.md)

## 1. Инициализация и состояние

- Модуль приложения `exts/client-mcp/src/Configuration/ManagedApplicationModule.bsl` содержит только экспортную глобальную переменную состояния `Мсп_Состояние` и обработчики событий приложения.
- Доступ к глобальному состоянию выполняется только через `exts/client-mcp/src/CommonModules/Мсп_СостояниеКлиент/Module.bsl`.
- Инициализация MCP-контура выполняется по цепочке `ПриНачалеРаботыСистемы -> Мсп_Сервер.Инициализировать() -> Мсп_РеестрКлиент.ИнициализироватьРеестры() + Мсп_ПровайдерыКлиент.ЗарегистрироватьПровайдеров()`.
- Состояние сервера обязательно содержит структуру `Реестры` и `ПараметрыЛогирования`.

## 2. Реестры и провайдеры

- В системе существуют ровно четыре базовых реестра: `Инструменты`, `Ресурсы`, `ШаблоныРесурсов`, `Промпты`.
- Регистрация объектов идет через строго типизированные дескрипторы в `Мсп_РеестрКлиент`; обязательные поля и допустимые типы проверяются при регистрации.
- Все дескрипторы обязаны хранить `ИсточникРегистрации`; для ресурсов, шаблонов и промптов также сохраняется `Провайдер`.
- Провайдеры обнаруживаются по метаданным: базовая подсистема `Мсп_Провайдеры` и подсистемы с суффиксом `_Мсп_Провайдеры`.
- Поддерживаемый тип провайдера в текущем ядре только один: `ОбщийМодуль`. Несовместимая сигнатура или невозможность создать модуль провайдера не лечится fallback-логикой и считается ошибкой регистрации.
- Включение и выключение объектов в рантайме делается через поле `Включён`; фильтрация по активным объектам остается обязанностью реестра.

Подтверждение: `exts/client-mcp/src/CommonModules/Мсп_МетаданныеСервер/Module.bsl`, `exts/client-mcp/src/CommonModules/Мсп_ПровайдерыКлиент/Module.bsl`, `exts/client-mcp/src/CommonModules/Мсп_РеестрКлиент/Module.bsl`, `tests/src/CommonModules/ОМ_Провайдеры/Module.bsl`, `tests/src/CommonModules/ОМ_РеестрКлиент/Module.bsl`.

## 3. Протокол и ответы

- Источником внешних событий считается только `WebTransport`; события от других источников игнорируются.
- Поддерживаемый набор событий транспорта зафиксирован в `Мсп_ТранспортСобытияКлиент`: `MCP_TOOL_CALL`, `MCP_RESOURCE_READ`, `MCP_PROMPT_GET`, `MCP_NOTIFICATION`, `MCP_RESOURCE_SUBSCRIBE`, `MCP_RESOURCE_UNSUBSCRIBE`, `MCP_TASK_CANCELLED`.
- Инструменты поддерживают только режимы выполнения `sync` и `task`.
- Флаг `ОтложенныйОтвет` допустим только для `sync`-инструментов; это специальный случай обычного `tools/call`, а не отдельный публичный execution mode.
- Внутренний запрос обработчика инструмента имеет стабильный контракт: `Аргументы`, `Контекст`, `Идентификатор` и опционально `ИдентификаторЗадачи`.
- Результаты публикуются через явные envelope:
  - инструмент: `result.content`
  - ресурс: `result.contents`
  - промпт: `result.messages`
- Сырые строковые и структурные результаты нормализуются на границе протокола в MCP-compatible ответ, но входные аргументы не нормализуются "на всякий случай": неверные типы приводят к `INVALID_PARAMS`.
- Явные доменные коды ошибок сохраняются в ответах; для JSON-RPC действует маппинг как минимум для `INVALID_PARAMS -> -32602`, `NOT_FOUND -> -32601`, прочих ошибок выполнения -> `-32603`.

Подтверждение: `exts/client-mcp/src/CommonModules/Мсп_ТранспортСобытияКлиент/Module.bsl`, `exts/client-mcp/src/CommonModules/Мсп_ПротоколКлиент/Module.bsl`, `exts/client-mcp/src/CommonModules/Мсп_Результаты/Module.bsl`, `tests/src/CommonModules/ОМ_ПротоколКлиент/Module.bsl`.

## 4. Границы между `client-mcp` и `test_client`

- `exts/client-mcp` владеет транспортом, жизненным циклом сервера, общим протоколом, состоянием, логированием и реестрами.
- `exts/test_client` владеет запуском/подключением тест-клиента, разрешением UI-объектов, построением UI-снимков и инструментами взаимодействия с интерфейсом.
- `test_client` интегрируется в ядро только как набор провайдеров, а не через специальные внутренние расширения транспорта.

Подтверждение: `exts/client-mcp/src/CommonModules/*`, `exts/test_client/src/CommonModules/*`, `tests/src/CommonModules/ОМ_Провайдеры/Module.bsl`.

## 5. URI и модель UI

- Поддерживаемые схемы URI ограничены `window://`, `form://`, `control://`, `snapshot://`.
- `window://active` является зарезервированным URI активного окна и не должен менять смысл.
- Сегменты URI кодируются percent-encoding поверх UTF-8; безопасные ASCII-символы не кодируются.
- `control://` всегда содержит два непустых сегмента: идентификатор формы и идентификатор элемента, разделенные `/`.
- Разрешение `window://...`, `form://...`, `control://...` идет строго по URI; при отсутствии объекта возвращается `NOT_FOUND`, при неоднозначности - `AMBIGUOUS_OBJECT`, а не "лучшее предположение".
- `snapshot://active` читает активное окно; `snapshot://{window_uri}` читает снимок окна по декодированному URI окна.
- `form://{name}` и `control://{form}/{name}` работают только для именованных сущностей. Пустой или неподдерживаемый URI считается ошибкой параметров.
- Операция `close` не применяется к `control://` URI.

Подтверждение: `exts/test_client/src/CommonModules/Мсп_URIКлиент/Module.bsl`, `exts/test_client/src/CommonModules/Мсп_РесурсыТестКлиент/Module.bsl`, `exts/test_client/src/CommonModules/Мсп_ФормыКлиент/Module.bsl`, `exts/test_client/src/CommonModules/Мсп_ОкнаКлиент/Module.bsl`, `tests/src/CommonModules/ОМ_URIКлиент/Module.bsl`, `tests/src/CommonModules/ОМ_ФормыКлиент/Module.bsl`, `tests/src/CommonModules/ОМ_ОкнаКлиент/Module.bsl`.

## 6. Состояние тест-клиента

- Память тест-клиента хранит как минимум `ТестКлиент`, `Подключен`, `ПоследняяОшибка`, `Порт`, `КомандаЗапуска`.
- Успех запуска тест-клиента определяется фактом подключения, а не только созданием процесса.
- Если тест-клиент не подключен, UI-операции и UI-ресурсы возвращают `NOT_READY` с явным текстом ошибки вместо пустых ответов.

Подтверждение: `exts/test_client/src/CommonModules/Мсп_УправлениеТестКлиентом/Module.bsl`, `exts/test_client/src/CommonModules/Мсп_ОкнаКлиент/Module.bsl`, `tests/src/CommonModules/ОМ_Провайдеры/Module.bsl`, `tests/src/CommonModules/ОМ_УправлениеТестКлиентом/Module.bsl`, `tests/src/CommonModules/ОМ_ПротоколКлиент/Module.bsl`.

## 7. Правила изменения

- Новую функциональность сначала добавлять как провайдер или развитие существующего провайдера. Изменение ядра `client-mcp` допустимо только если меняется общий контракт, состояние или транспорт.
- Любая скрытая нормализация входных параметров публичного API считается нарушением текущего контракта.
- Новые схемы URI, новые execution mode, новые типы транспортных событий или изменение структуры envelope ответов требуют отдельного архитектурного решения.
- При изменении инвариантов обновлять не только документацию, но и покрывающие тесты.
