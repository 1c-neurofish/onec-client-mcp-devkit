# Задача 05: Работа с формой

Статус: `done`

## Цель
Добавить URI-ориентированные действия над формой и её элементами, чтобы из MCP можно было эмулировать базовые пользовательские действия: перейти к элементу, нажать, ввести значение, выполнить выбор и корректно обработать открывшееся модальное окно.

## Область
Входит:
- Контракт инструментов действий по `window://...`, `form://...`, `control://...`.
- Инструмент `click(uri)` для кнопок и других кликабельных элементов.
- Инструмент `input(uri, text)` для полей ввода.
- Инструмент `select(uri, value?, index?)` для управляемого выбора значения в поддержанных элементах по правилу: ровно один из аргументов `value` или `index`.
- Единый post-action результат с возвратом активных `window_uri` и `form_uri`.
- Детект открытия модального окна после действия.
- YAxUnit-тесты с использованием реального тест-клиента для успешных и отрицательных сценариев.

Не входит:
- Drag-and-drop, горячие клавиши и составные пользовательские сценарии.
- Работа с файловыми диалогами и буфером обмена.
- Массовые действия по таблице, мультивыбор и сложные редакторы.
- Нотификации об изменении UI.
- Политика подтверждения опасных действий.

## Входные данные и допущения
- URI `window://...`, `form://{name}`, `control://{form}/{name}` уже являются публичным контрактом.
- `activate(uri)` из задачи 03 остаётся отдельным инструментом, но может использоваться внутри action-сценариев.
- На первом этапе поддерживаются только синхронные действия над одним объектом.
- Если после действия открывается модальное окно, в результате возвращаются URI нового активного окна и формы.
- Для `select` на первом этапе обязательно передаётся ровно один аргумент выбора: `value` или `index`.
- Для проверки сценариев доступен запуск YAxUnit на реальном тест-клиенте, а не только на моках и структурных заглушках.

## Подзадачи
1. Зафиксировать контракт инструментов `click`, `input`, `select` в API-документации и общем плане.
2. Добавить общее разрешение и валидацию объекта действия по URI:
   - объект найден однозначно;
   - объект видим и доступен для действия;
   - для `input` и `select` объект не находится в `readOnly`.
3. Реализовать `click(uri)` для поддержанных кликабельных элементов.
4. Реализовать `input(uri, text)` для полей ввода с возвратом итогового значения.
5. Реализовать `select(uri, value?, index?)` для поддержанного минимального сценария выбора с валидацией правила "ровно один из `value` или `index`".
6. Добавить post-action снимок активного окна и формы:
   - вернуть `window_uri` и `form_uri`;
   - признак `modal_opened`, если действие переключило фокус на модальное окно.
7. Обновить документацию и реализовать YAxUnit-тесты с использованием реального тест-клиента для успешных и отрицательных сценариев.

## Формат результата
- `click` возвращает `uri`, `type`, `title`, `name`, `window_uri`, `form_uri`, `modal_opened`.
- `input` возвращает `uri`, `type`, `title`, `name`, `value`, `window_uri`, `form_uri`, `modal_opened`.
- `select` возвращает `uri`, `type`, `title`, `name`, а также `value` или `index` выбранного варианта, `window_uri`, `form_uri`, `modal_opened`.

## Обработка ошибок
- `-32602` при некорректных параметрах, неподдерживаемом URI или нарушении правила `select`: ровно один из `value` или `index`.
- `UI_NOT_READY` при отсутствии подключения к тест-клиенту.
- `UI_NOT_FOUND` при отсутствии объекта по URI.
- `UI_AMBIGUOUS_OBJECT` при неоднозначном разрешении URI.
- `ACTION_DENIED` при попытке выполнить действие над неподдерживаемым, отключённым или `readOnly`-объектом.
- `-32603` при ошибке выполнения действия средствами тест-клиента.

## Проверки
- `click` выполняет действие по URI и возвращает актуальный `window_uri`.
- `input` изменяет значение поля и возвращает его в результате.
- `select` поддерживает минимальный сценарий выбора и валидирует отсутствие либо конфликт аргументов `value` и `index`.
- При открытии модального окна результат действия указывает на новое активное окно и форму.
- URI, не разрешившийся в объект, возвращает `UI_NOT_FOUND`.
- Неоднозначный URI возвращает `UI_AMBIGUOUS_OBJECT`.
- Неподдерживаемые типы объектов и недоступные элементы отклоняются с контрактной ошибкой.
- YAxUnit-сценарии выполняются через реальный тест-клиент и подтверждают базовые пользовательские действия без использования моков UI.

## Критерии готовности
- Инструменты `click`, `input`, `select` опубликованы в публичном контракте.
- Результат каждого действия пригоден для продолжения сценария по URI без повторного discovery.
- Модальное окно после действия различимо по результату инструмента.
- Отрицательные сценарии покрывают ошибки параметров, подключения, разрешения URI, типа элемента и состояния доступа.
- Для задачи добавлены YAxUnit-тесты на реальном тест-клиенте, покрывающие как минимум `click`, `input`, `select` и сценарий с модальным окном.

## Риски и неизвестные
- Методы тест-клиента для клика, ввода и выбора могут различаться по типам элементов формы.
- Для таблиц и полей выбора может понадобиться более узкий первый поддержанный сценарий, чем общий `select`.
- Детект нового модального окна может зависеть от таймингов перерисовки UI.
