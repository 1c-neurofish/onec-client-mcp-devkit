# Анализ exts/client-mcp/src на соответствие codex-rules-1c.md

## 1. SRP — Single Responsibility Principle

**Соответствует:** Большинство модулей имеют чётко определённую ответственность:
- `Мсп_СостояниеКлиент` — управление глобальным состоянием
- `Мсп_СериализацияКлиент` — JSON-сериализация
- `Мсп_Коллекции` — утилиты работы с коллекциями
- `Мсп_URIКлиент` — построение и разбор URI
- `Мсп_РеестрКлиент` — реестры инструментов/ресурсов/промптов
- `ManagedApplicationModule.bsl` — только глобальные переменные и обработчики событий приложения ✓

**Нарушения:**

- `Мсп_ФормыКлиент` (~1050 строк) — «god module». Совмещает: формирование снимков форм, поиск объектов по URI, активацию, закрытие, нажатие, ввод текста, выбор значений, навигацию по дереву объектов. Кандидат на декомпозицию минимум на 2-3 модуля (например: снимки, навигация/поиск, действия).

- `Мсп_ТранспортКлиент` (~558 строк) — совмещает управление компонентой, обработку событий, синхронизацию реестров и отправку ответов.

**Процедуры-кандидаты на декомпозицию (>40-60 строк):**
- `СинхронизироватьРеестрыСКомпонентой()` в `Мсп_ТранспортКлиент` — ~88 строк
- `ВыбратьПоURI()` в `Мсп_ФормыКлиент` — ~66 строк
- `ОткрытьФормуПоКоманде()` в `Мсп_ФормыКлиент` — ~60 строк
- `ОбработатьВнешнееСобытие()` в `Мсп_ТранспортКлиент` — ~56 строк
- `ЗарегистрироватьИнструменты()` в `Мсп_ИнструментТестКлиент` — ~49 строк

## 2. OCP — Open/Closed Principle

**Соответствует:** Провайдерная модель через подсистему `Мсп_Провайдеры` — отличная точка расширения. Новые инструменты/ресурсы добавляются через новые провайдеры без изменения ядра.

**Нарушения — длинные цепочки `Если/ИначеЕсли`:**

- `ТипПлатформыПоСтроке()` в `Мсп_ФормыКлиент` (строка 992) — 7 веток маппинга строк в типы. Заменить на `Соответствие`.
- `ТипОбъектаИнтерфейса()` в `Мсп_ФормыКлиент` (строка 1014) — 7 веток маппинга типов в строки. Заменить на `Соответствие`.
- `ОбработатьВнешнееСобытие()` в `Мсп_ТранспортКлиент` (строка 81) — 4 ветки по типу события. Заменить на таблицу обработчиков.
- `ПриоритетУровняЛога()` в `Мсп_ЛогированиеКлиент` (строка 208) — 4 ветки. Заменить на `Соответствие`.

## 3. DRY — Don't Repeat Yourself

**Нарушения:**

1. **Паттерн действия над объектом по URI** повторяется в `НажатьПоURI()`, `ВвестиТекстПоURI()`, `ВыбратьПоURI()`, `АктивизироватьПоURI()` в `Мсп_ФормыКлиент`: получить объект → проверить ошибку → проверить тип → проверить доступность → получить контекст до → выполнить действие → получить контекст после.

2. **Безопасный доступ к свойствам** через `Попытка/Исключение` + `Мсп_Коллекции.ПолучитьСвойство` повторяется 10+ раз в `Мсп_ОкнаКлиент` (строки 151-181, 216-233) и `Мсп_ФормыКлиент` (строки 1035-1049). Вынести в общую утилиту `БезопасноеСвойство(Объект, Имя, ЗначениеПоУмолчанию)`.

3. **Синхронизация реестров** в `СинхронизироватьРеестрыСКомпонентой()` в `Мсп_ТранспортКлиент` — 4 почти идентичных блока регистрации (инструменты, ресурсы, шаблоны ресурсов, промпты). Вынести в общий метод с параметрами.

4. **Обработка событий** в `ОбработатьToolCall()`, `ОбработатьResourceRead()`, `ОбработатьPromptGet()` в `Мсп_ТранспортКлиент` — одинаковая структура: подготовить запрос → извлечь параметры → вызвать протокол → отправить ответ.

5. **Построение JSON-схем** в `Мсп_ИнструментТестКлиент` (строки 341-433) — 7 функций с повторяющейся структурой. Вынести конструктор схемы.

6. **Извлечение параметров логирования** дублируется между `ПрименитьПараметрыЗапуска()` и `ПолучитьПараметрыЛогирования()` в `Мсп_ЛогированиеКлиент` (строки 170-183 повторяют 96-109).

## 4. YAGNI — You Aren't Gonna Need It

**В целом соблюдается.** Незначительное замечание:
- `ДекодироватьURL()` в `Мсп_URIКлиент` обрабатывает 5- и 6-байтовые UTF-8 последовательности (строки 300-307), которые выходят за пределы стандарта Unicode (максимум 4 байта).

---

## 5. Separation of Concerns

**Соответствует:**
- Форма `Мсп_УправлениеMCP` корректно делегирует логику сервисным модулям, не содержит бизнес-логики.
- Модуль приложения содержит только переменные и обработчики событий.

**Нарушения:**
- `Мсп_ФормыКлиент` смешивает логику UI-взаимодействия (нажатие, ввод, выбор) с логикой структуры UI (снимки, описания, навигация по дереву).

---

## 6. High Cohesion / Low Coupling

**Соответствует:** Модули взаимодействуют через минимальный публичный API. Зависимости направлены в одну сторону (инструменты → сервисы → утилиты).

**Нарушения:**
- `Мсп_ФормыКлиент` — низкая связность из-за слишком широкой ответственности.

---

## 7. Law of Demeter

**Нарушения:**
- `Module.bsl` формы `Мсп_УправлениеMCP` (строка 133): `Дескриптор.Описание.description` — цепочка доступа к вложенному свойству.
- `Мсп_ИнструментИнфоИБКлиент` (строка 97): `Расширение.ЗащитаОтОпасныхДействий.ПредупреждатьОбОпасныхДействиях` — глубокая цепочка.

---

## 8. Конструкторы структур (SKILL.md)

Правило: «Use constructor functions for all returned Структура values (no inline Новый Структура(...) outside constructors)».

**Нарушения — inline `Новый Структура(...)` вне конструкторов:**
- `Мсп_МетаданныеСервер.ОписаниеПровайдера()` (строка 26): `Новый Структура("Имя, Тип", Имя, Тип)`.
- `Мсп_ИнструментИнфоИБКлиент.СобратьИнформациюИБ()` (строки 49-52, 54, 57-60, 86, 92-97) — множественные inline `Новый Структура(...)` для вложенных данных.
- `Мсп_ТранспортКлиент.ПолучитьСводкуДанных()` (строка 292): inline `Новый Структура(...)`.
- `Мсп_ТранспортКлиент.ПодготовитьЗапросСобытия()` (строка 320): inline `Новый Структура(...)`.
